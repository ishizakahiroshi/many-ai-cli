# ai-cli-hub 実装規約・context分割・AI作業モデル

> 最終更新: 2026-05-07(木) 19:24:03

## context分割計画（plan_*.md）への対応

実装量が多い `plan_*.md` は **context（独立した実行単位）** を分けて実装する。各 context は `C1`, `C2` … の番号で管理し、プランファイルには「## context配分」セクションを必ず設ける。

ここでいう「context」は次のいずれかを指す（運用モードで使い分ける）：
- **subagent の独立 context**（標準運用：1指示で plan 全体を走り切るとき）
- **1個の会話セッション**（C 単位で手動指示するとき）

**適用範囲：** 実装計画だけでなく、調査メモ・検討メモ・将来計画など `plan_*.md` として作成するすべての `.md` ファイルに適用する。「まだ実装しない」内容でも context 配分を先頭に書いておくことで、後から実装に移行しやすくなる。

### plan_*.md を使うとき（実装時）

運用モードは2種類。原則は **A（subagent 委託・標準運用）**。

#### A. subagent 委託（標準・1指示で走り切る）

1. ユーザーは plan ファイルを指示するだけ（「この plan を実装して」等）
2. 親セッションは親 plan の `## context配分` 表だけ Read し、C1 から順に subagent へ委託する
3. 各 subagent は親plan path・担当C番号・子plan path（あれば）を受け取り、その C のみ実装する
4. 完了したら subagent が `## context配分` 表の該当 C を `plan` → `fix` に更新する
5. subagent は `完了/未完了`・`変更ファイル一覧`・`検証結果1行` の3点のみ親へ返す（調査ログ・思考過程は返さない）
6. 並列OKのCは1メッセージで複数 subagent を同時起動する

**自走運用の原則（subagent 委託・C 単位手動の両方に適用）：**

- **親 plan 1つを指示すれば走り切ることが目標**。ユーザー確認待ちを挟まず C 単位で進める（C 完了 → 次 C 自動進行）
- plan に書かれている作業は「ユーザー承認済み」とみなす。permission prompt が出ても止めず、plan 内記載と一致する操作なら許可して続行する
- Context・トークン消費を抑えるため、調査・確認の重複を避ける（同じファイルを複数回 Read しない、grep 結果を再取得しない）
- **停止条件は次の3つのみ：**
  1. plan 記載と矛盾する破壊的変更が必要になった
  2. 外部依存の致命的障害（ビルド不能・PTY ライブラリの不適合など plan 内で解決不能）
  3. 重大なテスト退行・設計書（`docs/ai-cli-hub-design-v0.1.0.md`）と矛盾する実装が必要になった
- 上記以外（軽微な迷い・選択肢の優劣判断・命名揺れ等）は plan 記載 or 既存実装に倣って自走する。確認に倒さない
- **subagent 種別の使い分け：** 実装は `general-purpose`、設計検討が必要なら `Plan`、軽い調査は `Explore` を使い分ける
- **親 → ユーザーへの最終報告**はグローバル `~/.claude/CLAUDE.md` の「ターン終端の出力ルール」（変更ファイル／稼働モデル／次のアクション）に従う（subagent → 親への内部報告である3点軽量報告とは別物）

#### B. C 単位の手動指示（小規模・対話的に進めたいとき）

1. セッション開始時にプランファイルを読み込み、**今回担当するcontext番号（C1/C2 等）を確認**
2. そのcontextの作業のみを実装。他のcontextには手を付けない
3. 実装完了後、「context配分」セクションの該当contextを `plan` → `fix` に更新
4. ユーザーが「C1 を実装して」と言った場合は、C1 の作業を実装

### plan_*.md を作成するときのcontext設計

**context番号：** `C1`, `C2`, `C3` … 形式で、常に純粋な連番（"Context 1" の略）。C番号は実行順を表す。

- **重要：`C2a`, `C2b` のような細分化表記は禁止**
- context を分割する必要があれば、C1, C2, C3... と番号を振り直す
- 理由：指示時に「C2 を実装して」で済み、`C2a` などの入力を避けるため
- 依存関係がある変更は、先に実施するcontextほど若い番号にする
- `[並列OK]` は「同時着手できる」という補足であり、C番号の実行順を崩す意味ではない
- **「Step 1 / Step 2」等の表記は使わない。必ず C番号で管理する**
- **contextの種別は `plan` / `fix` の2値のみ。`verify` `survey` `audit` 等の規約外種別は使わない**
- **context完了時は `## context配分` 表の該当行を `plan` → `fix` に更新する。「実行結果」セクションへの完了記述だけでは不足**
- 各 C 完了時の報告は最小化し、`fix:完了/plan:未完了`・`変更ファイル一覧`・`検証結果1行` の3点のみ返す
- 子計画のファイル名は親計画を作成した年月日時分秒と親計画番号を入れる（例：`親ファイル名_c1_overview.md`）

**分割基準：**
- 1contextに詰め込みすぎない。実装量・読み込みファイル数・調査コストを見積もり、余裕を持たせる
- 独立性の高い変更（別ファイル・別機能）は同一contextにまとめて並列効率を上げる
- 依存関係がある変更（後工程が前工程の成果物に依存）は別contextに分ける
- 調査・設計と実装は原則として分けない（同一contextで完結させる）

**ai-cli-hub 固有の分割例：**
- C1: `internal/proto/messages.go` 定義（Hub・ラッパー・UI が依存）
- C2: `internal/hub/server.go` HTTP+WS 起動 [C1 完了後]
- C3: `internal/wrapper/wrapper.go` PTY ラップ + 接続 [C1 完了後、C2 と並列OK]
- C4: `web/src/` Vue UI 骨組み [C1 と TypeScript 型同期、C2/C3 と並列OK]

**親子計画：**
- 大規模計画は、親 `plan_*.md` をハブにして、各contextの詳細を子 `plan_*.md` に分割してよい
- 1つの context の説明・対象ファイル・完了条件・判断ログが長くなる場合は、親 plan には概要と子 plan path のみを書き、詳細は対応する子 `plan_*.md` に分離する
- 親 plan の `## context配分` 表には、子 plan がある context の行に子 plan path を明記する
- 親子関係は1階層まで。子計画からさらに孫mdを作らない
- 親計画には子計画一覧と実行順序を置く
- 親計画の「実行順序」には `C1 → C2 → (C3, C4) → C5` のように矢印で順番を明示し、並列OKのcontextは括弧に入れる
- 子計画のファイル名・H1・context番号は、親のcontext番号と対応させる
- 子計画内の `context配分` は、その子計画の内部チェックリストであり、さらに別mdへ分割するための番号ではない
- 追加調査、対象ファイル、完了条件、判断ログは該当する子計画mdへ追記する

**記載方法：**
- 「## context配分」セクションに各contextの担当作業と状態（`plan` / `fix`）を記載
- **「## context配分」セクションはファイル先頭（H1見出し直下、概要より前）に配置**
- **並列実行可能なcontextは `[並列OK]` を明示**

**並列OK条件：**
- ①編集ファイルが重複しない
- ②一方の成果物を他方が参照しない
- ③共有リソース（WSメッセージ型・設定スキーマ等）の同時変更でない

**例：**
- 並列OK: 「C3とC4は別ファイル・独立機能のため並列実行可」
- 並列NG: 「C3はC1の `proto/messages.go` 定義を前提とするため直列」

## AI作業モデル使い分け

モデルは Agent ツール経由で指定。

- **計画・設計・立案：** `claude-opus-4-7` のサブエージェントに委ねること（Codexは適宜判断）
- **実装・コーディング：** `claude-sonnet-4-6` で直接実施（Codexは適宜判断）
- **簡単な調査・bash確認：** `claude-haiku-4-5-20251001` のサブエージェント（ls, cat, grep, find 等。メインcontext節約。複数コマンドはまとめて効率化）
- **単純な Edit/Grep/Glob/Read：** ツールを直接実行（モデル消費なし）

**作業規模に応じた判断：**
- 小さな修正・1〜2ファイルで完結する変更・調査系 → **自分で直接実施**
- 複数モジュールに跨る実装、広範なリファクタ、並列実行したい調査 → **サブエージェントに委託**
- 複数の独立タスク → **サブエージェントを並列起動して効率化**

## 作業進め方

- 大きめの作業はタスクを分割してPDCAで進める（TodoWriteで進捗管理）
- 作成したプランファイルは `docs/` 配下の適切なサブディレクトリに保存
- ソースコード修正前に develop ブランチにいることを確認（main にいる場合は develop へ切り替え）
  - 本リポジトリはまだ Git 初期化されていない可能性あり。`git status` で初期確認
- 複数の修正が並列で必要な場合は `develop-機能名` のブランチを作成可（コンフリクト注意）
- developブランチからmainへのマージはユーザーが実施

## docs ファイル配置・命名規則

### ファイル置き場

| ディレクトリ | 内容 |
|---|---|
| `docs/local/` | GitHub 非公開ファイル（plan, session, reference 等） |
| `docs/local/archive/` | 対応完了済み・旧版ドキュメント置き場 |

> 規模が拡大したら `docs/setup/` `docs/manual/` 等のサブディレクトリを追加する。最初は `docs/` 直下で十分。

### 命名規則（`.md` ファイルのみ対象）

| プレフィックス | 用途 |
|---|---|
| `plan_*.md` | 計画、対応中（`docs/` に配置） |
| `pending_*.md` | 対応が保留している |
| `setup_*.md` | 初回構築・環境設定（再実行が想定されない一回作業） |
| `manual_*.md` | 繰り返し参照する運用手順 |
| `reference_*.md` | 実装時に繰り返し参照する定義・一覧（更新義務あり） |
| なし | 記録用の通常ドキュメント |
| （完了済み） | ファイル名を変えず `docs/archive/` に移動 |

**setup と manual の区別：** 「1回やったら終わり」は `setup_`、「再実行・参照が想定される」は `manual_`

### 作業完了時のファイル移動

`plan_*.md` の作業が完了したら：
1. `python C:\dev\tools\dev-scripts\check_archivable_plans.py` で対象を確認
2. ユーザーに `docs/local/archive/` への移動を確認（`Y:1/N:0`）
3. 承認を得たら `python C:\dev\tools\dev-scripts\check_archivable_plans.py --move` を実行
4. **ファイル名は変えない**
5. 移動後は、他ファイルからの相対リンクを grep で機械的に張り直す

### 最終更新日時の記載

`docs/` 配下と `CLAUDE/*.md` のすべての `.md` ファイルは、H1見出しの直後に `> 最終更新: YYYY-MM-DD(曜) HH:MM:SS` を記載。

新規作成・更新時は、記入直前に PowerShell で実行：
```powershell
$d = Get-Date; "{0}({1}) {2}" -f $d.ToString("yyyy-MM-dd"), "日月火水木金土"[$d.DayOfWeek.value__], $d.ToString("HH:mm:ss")
```

その出力値をそのまま使用。bash の場合は `date` が英語曜日で出るため手動で日本語曜日に変換すること。
