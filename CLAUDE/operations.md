# many-ai-cli Git・コミット・出力ルール

> 最終更新: 2026-06-07(日) 12:57:07

## Git コミット

### コミット実施主体

**原則：** Git コミットはユーザーが実施する
- AI が編集・実装を完了 → 変更ファイル一覧を報告 → ユーザーが `git add` と `git commit` を実行

**例外：** AI エージェント（Codex 等）に「コミットしてください」と明示的に指示された場合のみ、Claude / Codex が `git commit` を実行
- この場合、実行前にコミット内容の確認を促す
- コミットメッセージは本ガイドに従う

> 本リポジトリは現状 Git 未初期化の可能性あり。最初の `git init` 〜 `.gitignore` 整備はユーザー実施を想定。

### コミットメッセージ

- **言語：** 必ず日本語で記述
- **prefix：** 英語のままで OK（feat:, fix:, refactor:, docs:, chore:, test:, build: など）
- **例：**
  - `feat: Hub の WebSocket エンドポイントを追加`
  - `fix: PTY 終了時に子プロセスが孤児化する問題を修正`
  - `docs: 設計書 v2 を追加し v1 を archive へ移動`
- **AI作成の場合：** 末尾に `Co-Authored-By: Claude <model> <noreply@anthropic.com>` を付与

## セキュリティ

設計書 §8（ローカルサーバの設計上の配慮）と §21-5（ローカルサーバが立つことの透明性）に準拠：

- **バインドは `127.0.0.1` 固定**
- **ランダムトークンを起動時生成し URL に付与**（`?token=xxx`）
- **外部ネットワーク通信は必要な用途に限定する**（ユーザー操作、公式ドキュメント取得、CLI 本来の通信など）。意図しない自動送信やテレメトリは追加しない
- **設定ファイルは本人読み書きのみ**（Unix: `0600`）
- **PTY ログにシークレットが入りうる前提**で、ログディレクトリも本人のみ閲覧可

## 外部への情報公開（GitHub Issue / フォーラム / Gist 等）

GitHub Issue・フォーラム・外部サービスへ投稿する際は、以下の情報が含まれていないか必ず確認してから送信すること。

### 含めてはいけない情報

- 個人ユーザー名・メールアドレス（個人の実アドレス。具体例も本ファイルに書かない）
- ローカル PC のフルパス（`C:\Users\<user>\...` 等。`~/` や `<HOME>/` に置換）
- 開発中の機密パターン（実環境固有のコマンド・社内プロジェクト名など）
- スクリーンショットに映り込んだ個人名・社内固有の名称（タスクバー・ウィンドウタイトル含む）

### チェック手順

1. 本文中の固有名詞を検索し、仮名または汎用名（`example`, `user123`, `~/project` 等）に置き換える
2. 添付スクリーンショットを目視確認し、個人名・パス・社内名称が映り込んでいないか確認する

## 出力ルール（ターン終端）

編集を伴うターンの末尾に必ず以下3点を出力する。利用中 AI のグローバル指示に別形式がある場合は、同じ情報を保ったままその形式に合わせる。

```
**変更ファイル**: <path1>, <path2>, ...（編集なしなら「なし」）
**稼働モデル**: <モデル名>（subagent 委託時はそれぞれのモデルも併記）
**次のアクション**: <ユーザーが次にやること1行.なければ「なし」>
```

- 読み取りのみ・調査のみ・質問応答のみのターンでは出さない
- 「次のアクション」は具体的に書く
