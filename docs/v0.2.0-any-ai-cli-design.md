## 1. v0.2.0 の位置づけ

v0.2.0 は v0.1.3 からの minor release。テーマは「Hub UI を単なる承認ビューから、複数セッション運用の作業面へ拡張する」こと。

主な変化:

- Files / Git / Chat / Split / Multi を統合タブバーへ追加
- Claude / Codex の spawn 時モデル選択を Ollama route まで拡張
- approval detection を remote profile 方式へ移行
- UI 設定を localStorage 依存から `config.yaml:user_prefs` へ移行
- WSL 内 Hub を Windows 側から起動する専用 launcher を追加
- `/api/info` の `env_*` により、Local / WSL / リモートサーバー / リモートサーバー（トンネル） を title・favicon・ヘッダー badge で識別できるようにした
- ログを人間が読みやすい transcript として再生成できるようにした

## 2. 公開スコープ

公開 README 上の wrap 対象は Claude Code / Codex CLI / GitHub Copilot CLI / Cursor Agent CLI。Gemini CLI は対象外。OpenCode は v0.2.0 では非公開・未実装扱い。Cursor Agent はサインイン済みの公式 `cursor-agent` CLI を PTY 起動するだけで、Cursor の認証情報には触れない。

Ollama は provider ではなく route / model backend。Claude Code / Codex CLI の子プロセスに環境変数を注入し、既存 provider CLI を Ollama 経由で使う。

検証状況:

- 実機検証済み: Windows
- WSL: Windows launcher `any-ai-cli-wsl.exe` 経由を v0.2.0 の公開導線に含める
- native Linux / macOS: build 対象だが十分な実機検証は未完了

## 2-1. Runtime / Environment Identity

Hub UI は `GET /api/info` を single source として実行環境を表示する。従来の `runtime_mode` / `runtime_label` / `ssh` / `host_ip` は後方互換のため残し、ブラウザタブ・favicon・ヘッダー badge の瞬間識別には `env_*` を使う。

`env_kind` の標準値:

| env_kind | 表示 | 用途 | favicon |
|---|---|---|---|
| `local` | Local | ローカル OS 上で直接起動した Hub | 緑 + `L` |
| `wsl` | WSL | WSL 内、または Windows launcher 経由の WSL Hub | 青 + `W` |
| `remote` | リモートサーバー | SSH 経由で起動したリモートサーバー上の Hub | オレンジ + `V` |
| `remote-tunnel` | リモートサーバー（トンネル） | SSH local forwarding 経由で常駐リモートサーバー Hub を操作 | 赤 + `T` |

判定優先順は `ANY_AI_CLI_ENV_KIND` → `config.yaml:hub.env_kind` → `/api/net-hint` → runtime / SSH 環境変数 → `local`。不正な `env_kind` は `local` へフォールバックする。`ANY_AI_CLI_HOST_LABEL` と `/api/net-hint.host_label` は `env_host_label` として `/api/info` に返し、リモートサーバー / tunnel の識別補助に使う。

`GET /api/info` の環境識別フィールド:

| フィールド | 例 | 内容 |
|---|---|---|
| `env_kind` | `remote-tunnel` | UI 表示用の環境種別 |
| `env_label` | `リモートサーバー（トンネル）` | ヘッダー / Settings 表示名 |
| `env_short` | `T` | favicon 1 文字表示 |
| `env_color` | `#ef4444` | favicon / badge 色 |
| `env_title` | `ANY-AI-CLI リモートサーバー（トンネル）` | ブラウザタブタイトル |
| `env_host_label` | `203.0.113.10` | リモートサーバー / tunnel の接続先ラベル |

## 4. Chat / Split

Chat はセッションごとに `Message[]` を in-memory で保持する。データ源は PTY stream と UI 操作イベント。

取り込むイベント:

- user input: 送信本文
- AI output: PTY output chunk を正規化して commit
- approval: 承認待ちと回答済み情報
- attach: 添付ファイル名、種別、保存 path、画像 preview URL

ターン境界は Hub から挿入される OSC marker `\x1b]47777;user-turn\x07` を使う。ブラウザ再接続時の PTY replay でも履歴を seed できるようにしているが、完全な jsonl 再構築は v0.2.0 の公開仕様ではない。

Chat UI:

- user / AI の吹き出し表示
- provider icon / user avatar
- message copy
- tool call の折りたたみ
- raw PTY text の確認
- filter: all / user / AI / attachment / approval
- search と current match 移動
- minimap による message jump

Split は terminal と chat を同時表示するモード。送信後や AI output 追記時は、terminal / chat それぞれの bottom state を見て自動追従する。

## 5. Multi タブ

Multi は複数セッションを同時に見るためのグリッドビュー。

設計:

- 最大 18 ペイン想定
- grid picker でレイアウト選択
- 各 slot に xterm container を attach
- active slot を sidebar / input / approval action-bar と同期
- session close pending 中の slot 再表示を抑止
- ResizeObserver で各ペインの terminal fit / resize を送信
- scrollback は通常 terminal より軽い値を使える

Multi 表示中も、入力欄・添付・approval action-bar は共有 UI のまま。フォーカス中ペインの session id に対して送信する。

## 5b. Detached Session Grid と Shell provider

v0.2.0 以降（plan_detached-session-grid.md / C1〜C6）で追加。

### Detached Session Grid

Hub 本体とは別ブラウザウィンドウで session を grid 表示する機能。session の生死・承認・終了は Hub が持つ。別窓は表示（xterm attach / resize / focus）だけを担当する。

URL 形式:

```
/?view=detached-grid&layout=2x2&session_ids=1,2,3,4&token=<token>
```

| パラメータ | 説明 |
|---|---|
| `view=detached-grid` | detached モードを判定するフラグ |
| `layout` | `1x1` / `1x2` / `2x2` / `2x3` / `3x3` / `4x3` / `6x3`。cols max=6, rows max=3 |
| `session_ids` | カンマ区切りの session ID 列。Hub 未登録 ID は pending 表示 |
| `token` | Hub 認証トークン |

実装: `web/src/app/detached-grid.ts` の `DetachedGridManager` と `initDetachedGridMode`。

Detached モードでは通常 Hub UI 要素（session-list / input-bar / action-bar / settings 等）を非表示にし、header に「⊞ Hub」ボタンを表示して Hub 本体に戻れるようにする。

### 新規セッション起動 UI の Detached 設定

`＋ 新しいセッション` パネル（`web/src/app/spawn-panel.ts`）に Detached 設定を追加:

| 項目 | 説明 |
|---|---|
| Open target | `Hub tab` / `Detached window` / `Attach to current grid` |
| Grid layout | `1x1` / `1x2` / `2x2` / `2x3` / `3x3` |
| Preset | `Claude + Shell 2x2` / `Shell 2x2` / `Shell 3x3` / `Selected sessions` / `Project sessions` など |

`Claude + Shell 2x2` preset を選んだ場合: AI session 1 枚 + Shell session 3 枚を spawn し、返ってきた session ID 群で Detached Grid URL を生成して `window.open()` で別窓を開く。

### Shell provider

`/api/spawn` の provider に `shell` を追加した。

| 項目 | 挙動 |
|---|---|
| provider 名 | `shell` |
| コマンド解決 | Windows: pwsh.exe → powershell.exe → COMSPEC → cmd.exe。Unix: $SHELL → bash → sh |
| AI 固有フラグ | model / route / permission-mode / approval-rules injection は全てスキップ |
| session card / Multi | `Shell` provider chip を表示 |

Shell は hub 管理下の通常 session type として扱われる。承認検出・AI チャット抽出・slash command picker・done summary push など AI 固有機能は shell session では無効化される。

### `POST /api/spawn-grid`

複数 session を一括起動して Detached Grid 用の layout 情報を返すエンドポイント。

request:

```json
{ "preset": "shell", "layout": "2x2", "count": 4, "cwd": "...", "label_prefix": "grid" }
```

response:

```json
{ "ok": true, "layout": "2x2", "count": 4 }
```

対応 preset: `shell`（Shell session を count 枚起動）、`ai+shell`（AI 1 枚 + Shell (count-1) 枚）。layout 省略時は count から自動算出（`calcGridLayout`）。count は 1〜18。

実装: `internal/hub/spawn_handler.go` の `handleSpawnGrid` / `calcGridLayout`。

## 6. Files タブ

Files は旧 docs browser を rename / 拡張した機能。公開 UI 上は project group の Files 導線、セッションカード右クリック、`Ctrl+Shift+F` から開く。

Backend API:

| API | 内容 |
|---|---|
| `GET /api/files-roots` | session / project に基づく root candidates |
| `GET /api/files-list` | root 以下の file tree flat list |
| `GET /api/files-content` | preview 用テキスト取得 |
| `GET /api/files-asset` | image / media preview asset |
| `GET /api/files-download` | file download attachment |
| `POST /api/files-move` | single / multi file move |
| `POST /api/files-rename` | file / directory rename |
| `POST /api/files-delete-dir` | empty directory delete only |
| `POST /api/files-save` | テキストファイルの上書き保存（競合検出付き） |

### POST /api/files-save

リクエスト body（JSON）:

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `path` | string | 必須 | 書き込み対象ファイルの絶対パス |
| `content` | string | 必須 | 保存するテキスト内容 |
| `baseMtime` | string (RFC3339) | 省略可 | 競合検出用の基準 mtime。指定時はサーバ側 mtime と照合 |

レスポンス body（JSON）:

| フィールド | 型 | 説明 |
|---|---|---|
| `ok` | bool | 成功時 `true` |
| `path` | string | 保存したファイルの絶対パス |
| `size` | int | 保存後のファイルサイズ（バイト） |
| `mtime` | string (RFC3339) | 保存後の mtime（次回保存時の `baseMtime` に使う） |
| `error` | string | エラーコード（失敗時） |
| `detail` | string | エラー詳細（失敗時） |

エラーコード一覧:

| HTTP | `error` | 意味 |
|---|---|---|
| 400 | `bad_request` | path 未指定 / 相対パス / ディレクトリ指定 / JSON 不正 |
| 401 | — | token 不一致 |
| 403 | `forbidden` | allowed roots 外 / 非テキスト拡張子 |
| 404 | `not_found` | ファイルが存在しない（新規作成不可） |
| 409 | `conflict` | `baseMtime` と現在 mtime が不一致。レスポンスに現在 `mtime` を含める |
| 413 | `too_large` | content が 1 MiB 超 / request body が 2 MiB 超 |
| 500 | `internal_error` / `write_failed` | stat / write 失敗 |

制約:

- traversal 防止のため allowed roots は cwd / git root に限定
- walk は depth / item count 上限あり
- symlink は追跡しない
- preview text は許可拡張子とサイズ上限を持つ
- directory delete は安全側に寄せ、空 directory 以外を拒否
- multi move は preflight 後 all-or-nothing。実行中失敗時は成功済み rename を逆順 rollback

Frontend:

- tree search
- Markdown / code preview
- image / media preview
- context menu: copy path, open, reveal, move, rename
- multi select + D&D / move button
- restored orphan tab の修復

## 7. Git タブ

Git タブは read-heavy の repository inspection UI。branch badge click、session card context menu、`Ctrl+Shift+G` で開く。

Backend API:

| API | 内容 |
|---|---|
| `GET /api/git-log` | commit list。`ref`, `limit`, `skip` 対応 |
| `GET /api/git-show` | commit detail / file diff |
| `GET /api/git-refs` | local / remote / tag / all refs |
| `GET /api/git-status` | working tree status |
| `POST /api/git-fetch` | guarded fetch |
| `POST /api/git-commit-message` | commit message generation helper |

## 8. Approval

### Multi-question approval

`[ANY-AI-CLI]` 承認ブロック内に複数の番号付き質問がある場合、Hub は stacked choices として表示し、選択結果をまとめて PTY へ送る。

主な仕様:

- question ごとに選択状態を保持
- progress / clear / keyboard navigation
- Submit all で space-separated digit string を送信
- global numbering で出力された選択肢も、UI 側で question 内 position へ補正
- false positive 時は banner dismiss が可能

### Go-side native approval detection

v0.2.0 では frontend buffer scan に加えて Go 側 VT buffer を使う native approval detection も導入。`internal/hub/approval_detector.go` / `vt_buffer.go` が terminal output を正規化して、Claude / Codex / Copilot / Cursor Agent の prompt shape を検出する（Cursor Agent のパターンは実機未確認のため暫定で Copilot 流用）。

検出時は WebSocket event:

- `approval_detected`
- `approval_cleared`

を UI に送り、UI 側 action-bar と状態表示へ反映する。

### Approval marker rule injection

承認ボタン機能が有効な場合、wrapped session の起動時に `[ANY-AI-CLI]` マーカー出力ルールを instruction file へ注入する。Claude と Codex は user/global instruction file、Copilot / Cursor Agent は project instruction root の `AGENTS.md` に共有ブロックを貼る。

| provider | 注入先 | 方式 |
|---|---|---|
| Claude | `~/.claude/CLAUDE.md` | `@~/.any-ai-cli/approval-rules.md` import 行 |
| Codex | `$CODEX_HOME/AGENTS.md`、未設定なら `~/.codex/AGENTS.md` | `<!-- any-ai-cli:approval-rules -->` 共有ブロック |
| GitHub Copilot | `<project instruction root>/AGENTS.md` | 共有ブロック |
| Cursor Agent | `<project instruction root>/AGENTS.md` | 共有ブロック |

`<project instruction root>` は git root を優先し、git root が取れない場合は session cwd。共有ブロックは path 単位で冪等注入し、そのファイルを読む active wrap が0になった時点で any-ai-cli のブロックだけ削除する。旧実装で Codex 用に使っていた `<project instruction root>/AGENTS.md` は cleanup 対象として残す。

### Approval pattern profiles

承認 trigger phrase は `resources/approval-patterns/{claude,codex,copilot,cursor-agent,common}.md` を source とし、Hub 起動時に fetch / cache する。

| ファイル | 役割 |
|---|---|
| `<provider>.official.json` | GitHub から同期した読み取り専用 profile |
| `<provider>.custom.json` | ユーザー編集用 profile |
| `<provider>.json` | active profile mirror。既存 frontend 互換 |

API:

- `GET /api/approval-patterns`
- `PUT /api/approval-patterns/<provider>`
- `GET/POST /api/approval-patterns/profile`
- `POST /api/approval-patterns/copy-official`

remote sync で差分があれば `approval_patterns_updated` を broadcast し、UI が toast / reload する。

## 9. Model Picker / Ollama Route

Model picker は `web/src/app/spawn-panel.ts` と `/api/models` で実装される。

Model groups:

| Group | provider | route | source |
|---|---|---|---|
| Anthropic | `claude` | `anthropic` | `resources/models/defaults.json` remote fetch + fallback |
| OpenAI | `codex` | `openai` | `resources/models/defaults.json` remote fetch + fallback |
| GitHub Copilot | `copilot` | （空） | `resources/models/defaults.json` remote fetch + fallback |
| Cursor Agent | `cursor-agent` | （空） | `resources/models/defaults.json` remote fetch + fallback |
| Ollama Cloud | both | `ollama` | local daemon `/api/tags` の `remote_host` 付き aliases |
| Ollama Local | both | `ollama` | local daemon `/api/tags` + `config.yaml:local_models` |

Copilot のモデル候補グループは route 空（env 注入なし）で表示する。選択・手入力いずれの model も `copilot --model <model>` へ素通しし、対応可否は GitHub Copilot CLI 側に委ねる（slug が外れても datalist の自由入力で補える）。候補は Copilot CLI の GA モデルファミリ（Claude / GPT / Gemini）を載せ、slug 不確実な Raptor mini は除外する。

Cursor Agent も同様に route 空（env 注入なし）で表示し、選択・手入力いずれの model も `cursor-agent --model <model>` へ素通しする。具体的なモデル候補は実機未確認のため暫定とする。

`/api/models` は local daemon 結果を 60 秒 cache し、default model list は remote fetch 失敗時に hardcoded fallback を使う。

Spawn 時の route 判定:

1. request body の `route` が明示されていれば採用
2. model が known Ollama IDs に含まれれば `ollama`
3. model 名に `:cloud` を含めば `ollama`
4. provider が Claude なら `anthropic`
5. provider が Codex なら `openai`
6. provider が Copilot なら `""`（env 注入なし）
7. provider が Cursor Agent なら `""`（env 注入なし）

Ollama route の env preset:

| provider | env |
|---|---|
| Claude | `ANTHROPIC_AUTH_TOKEN=ollama`, `ANTHROPIC_API_KEY=`, `ANTHROPIC_BASE_URL=http://localhost:11434` |
| Codex | `OPENAI_API_KEY=ollama`, `OPENAI_BASE_URL=http://localhost:11434/v1` |

Codex × Ollama では `--codex-oss` を wrapper 側へ渡す。Ollama route の model は `last_model` に保存しない。次回空欄 spawn が意図せず Ollama route に戻ることを防ぐため。

## 10. Server-side user preferences

v0.2.0 ではユーザー機能設定を `config.yaml:user_prefs` に移した。

代表項目:

- voice grace seconds
- auto-submit trigger
- notification sound
- approval auto-switch
- quick commands
- usage links
- favorites / group order / session order / project favorites
- cwd history
- spawn defaults / last model
- avatar / display name

Frontend は初回 load 時に `GET /api/user-prefs` で server state を mirror し、localStorage 既存値を一度だけ server へ migrate する。以降は `setUserPref()` が localStorage cache と server PUT を同時に行う。

## 11. WSL launcher

WSL 接続は統合ランチャー（§11b）の `wsl` プロファイルで提供する。実装は `internal/launcher/connector_wsl_windows.go` の `WSLConnector`。

> 旧 standalone launcher `cmd/any-ai-cli-wsl/`（`any-ai-cli-wsl.exe`）は v0.3.0 で削除した。同等機能は `WSLConnector` が引き継いでいる。

起動フロー:

1. Windows 側で port を選ぶ。47777 から 100 刻みで Windows-side listener を probe
2. `wsl.exe [-d distro] --cd <cwd> -- bash -ilc 'export ANY_AI_CLI_WSL_LAUNCHER=1; exec any-ai-cli serve --port <port>'`
3. WSL 側 stdout / stderr を scan し、banner の Hub URL を抽出
4. Windows の default browser を開く
5. launcher 終了時に同 port の WSL orphan `any-ai-cli serve` を best-effort cleanup

`bash -ilc` を使う理由は、Ubuntu 既定の `.bashrc` が非 interactive shell で早期 return するため。pnpm / nvm / cargo 等の PATH を読むために login + interactive を併用する。

`ANY_AI_CLI_WSL_LAUNCHER=1` のとき、Linux 側 Hub は log dir を Windows user profile 配下へ寄せる。Hub UI の open folder が Windows Explorer で扱いやすい path になるため。

## 11b. 統合ランチャー（any-ai-cli-launcher.exe）

`cmd/any-ai-cli-launcher/main_windows.go` は Windows 専用の統合ランチャー。WSL とリモートサーバー SSH の両プロファイルを扱う。

### バイナリ構成

| バイナリ | 役割 |
|---|---|
| `any-ai-cli-launcher.exe` | 統合ランチャー。wsl / ssh プロファイルを一元管理 |

> WSL 専用の旧ランチャー `any-ai-cli-wsl.exe` は v0.3.0 で削除（機能は wsl プロファイルへ統合済み）。

### プロファイルファイル

保存先: `~/.any-ai-cli/launcher-profiles.yaml`（`config.yaml` とは分離）。

```yaml
version: 1
last_used: my-wsl           # 直近使ったプロファイル名（launcher が自動更新）
profiles:
  - name: my-wsl
    type: wsl
    distro: Ubuntu-22.04    # 省略 = wsl.exe の既定ディストリビューション
    hub_port: 0             # 0 = Windows 側衝突を避けて自動選択
    cwd: /home/user/projects

  - name: my-remote
    type: ssh
    mode: serve             # serve（既定）= リモートで any-ai-cli serve を起動
    host: remote.example.com
    user: your-user
    ssh_port: 22            # 0 = 22 または ~/.ssh/config の値
    identity_file: ""       # 空 = ssh デフォルト / agent
    hub_port: 47777         # 0 = 自動選択
    cwd: /home/your-user/projects

  - name: remote-docker
    type: ssh
    mode: tunnel            # tunnel = 常駐 Hub へのポートフォワードのみ
    host: remote.example.com
    user: your-user
    hub_port: 47801         # 必須（0 は不可）。リモート Hub のポートと同じにする
    token_command: "docker exec any-ai-cli-user1 sh -c 'grep ^token ~/.any-ai-cli/config.yaml | cut -d\" \" -f2'"
```

フィールド詳細:

| フィールド | 説明 |
|---|---|
| `name` | プロファイル名（一意）。`--profile <name>` で指定できる |
| `type` | `wsl` または `ssh` |
| `mode` | SSH プロファイルのみ。`serve`（既定）または `tunnel` |
| `host` | SSH 接続先ホスト名（`~/.ssh/config` のエイリアス可。`user@host` 形式も可） |
| `user` | SSH ユーザー名（`host` に `user@` が含まれる場合はそちらが優先） |
| `ssh_port` | SSH ポート（0 = 22 または config のデフォルト） |
| `identity_file` | 秘密鍵パス（空 = ssh デフォルトまたは agent）。パスフレーズ・パスワードは保存しない |
| `hub_port` | Hub が listen するポート。`serve` モードでは 0 で自動選択可。`tunnel` モードは必須 |
| `cwd` | リモート側の作業ディレクトリ（WSL では WSL パス形式） |
| `distro` | WSL ディストリビューション名（省略 = wsl.exe の既定） |
| `binary` | リモート側の `any-ai-cli` バイナリ名（省略 = `any-ai-cli`） |
| `token_command` | `tunnel` モードのみ必須。SSH 経由でリモート実行し、stdout がトークン |

### CLI フラグ

```
any-ai-cli-launcher.exe [--profile <name>] [--last] [--ui]
```

| フラグ | 動作 |
|---|---|
| `--profile <name>` | 指定プロファイルで即接続 |
| `--last` | 前回使ったプロファイルで即接続 |
| `--ui` | 常に接続先選択画面を表示 |
| (なし) | プロファイル 1 件なら即接続。複数なら選択画面を表示 |

### serve モードの接続フロー

1. Windows 側でポートを選ぶ（`hub_port: 0` の場合は 47777 から 100 刻みで空きを probe）
2. `ssh.exe -t -L <port>:127.0.0.1:<port> [user@]host -- bash -ilc "any-ai-cli serve --port <port>"` を起動
3. stdout / stderr を scan し、バナーの Hub URL を抽出
4. URL 中のポートが指定値と一致しないとき（リモート側で自動移動した場合）は kill して次候補で再試行（最大 5 回）
5. Windows の既定ブラウザで Hub UI を開く
6. Ctrl+C でリモートの serve を cleanup してから終了

### tunnel モードの接続フロー

1. `ssh.exe -N -L <port>:127.0.0.1:<port> [user@]host` でポートフォワードのみ確立
2. `token_command` を SSH 経由で実行し、stdout をトークンとして取得（5 秒タイムアウト）
3. `http://127.0.0.1:<port>/api/info?token=<token>` に対して最大 20 回ポーリング（500 ms 間隔）
4. `/api/net-hint` に `ssh=true`, `host_label`, `env_kind=remote-tunnel` を best-effort で登録する
5. 疎通確認後にブラウザで `http://127.0.0.1:<port>/?token=<token>&via=ssh&host_label=<host>&env_kind=remote-tunnel` を開く
6. Ctrl+C でトンネルを閉じる（リモート Hub は停止しない）

tunnel モードでローカル側ポートが衝突した場合は `-o ExitOnForwardFailure=yes` で即失敗。`hub_port` は固定が前提のため +100 再試行はしない。

### セキュリティ前提

serve / tunnel いずれのモードでも、Hub のセキュリティモデルは変わらない。

- Hub は `127.0.0.1` bind 固定（`0.0.0.0` bind やリバースプロキシ経由の公開はしない）
- トンネルは `127.0.0.1` 同士の local forward のみ（`-g` や `GatewayPorts` は不使用）
- `token_command` の実行結果（トークン）は `launcher-profiles.yaml` に保存しない
- パスワード・パスフレーズは保存しない（`-o BatchMode=yes` で対話を禁止し、失敗時はエラー表示）

実装ファイル:

- `internal/launcher/profile.go` — プロファイル yaml スキーマ / 読み書き / バリデーション
- `internal/launcher/connector_wsl_windows.go` — WSL 接続フロー
- `internal/launcher/connector_ssh_windows.go` — SSH serve / tunnel 接続フロー
- `cmd/any-ai-cli-launcher/main_windows.go` — CLI エントリポイント（フラグ処理・プロファイル選択・接続）

## 17. Security / Privacy

v0.2.0 で host operation API が増えたため、localhost-only 前提がより重要になった。

守る前提:

- Hub は `127.0.0.1` bind 固定
- token なし API / WebSocket は既定で拒否。例外として `hub.allow_loopback_without_token: true` の場合のみ、RemoteAddr が loopback または `hub.trusted_networks` の CIDR に入るリクエストを許可する。`trusted_networks` は `/32` など最小範囲を前提とし、`0.0.0.0/0` / `::/0` は設定エラー
- `hub.allowed_hosts` は WireGuard などの私設経路用に Host / Origin 許可リストを拡張する設定。値は IP literal または hostname のみで、wildcard / port 付き指定は設定エラー
- `/api/open-dir` 等は loopback remote address を確認
- reverse proxy / `0.0.0.0` bind / remote server 運用は禁止
- session raw log / jsonl / transcript / attach は機密情報として扱う

Outbound traffic:

- slash command sources: GitHub raw
- approval pattern sources: GitHub raw
- model defaults: GitHub raw
- wrapped CLI traffic: Claude / Codex / Ollama 側の通常通信

`any-ai-cli` 自身は telemetry を送信しない。
