# ⚠️ 重要：セキュリティリスクを理解した上でご利用ください

**スマホ / リモートから Hub に接続する機能は、便利さと引き換えに重大なリスクを伴います。内容を理解・同意できる場合にのみ有効化してください。**

- **接続情報が漏れると、第三者があなたのマシンを遠隔操作できる。** many-ai-cli の Hub は AI CLI を通じてファイルの閲覧・改ざん・削除、任意コマンド実行、認証情報の抜き取りまで可能。接続手段（token / QR / SSH 鍵 / VPN 認証）の漏洩は、**最悪マシンの完全な乗っ取りに直結する**。
- **token を含む URL / QR はパスワードそのもの。** スクリーンショット共有・画面共有・SNS 投稿・第三者への提示は厳禁。**一度漏れたら即座に悪用され得る。**
- **SSH サーバーの公開・VPN の設定・`allowed_hosts` / `trusted_networks` の変更は、すべて利用者自身の責任。** 設定を誤ると**意図せず全世界へマシンを露出**させる危険がある。
- 接続は**信頼できる回線・端末からのみ**。**公共 Wi-Fi（カフェ・空港・ホテル等）は盗聴・中間者攻撃の危険が高く、絶対に使わないこと。** 他人の端末での利用も避ける。
- 本ソフトウェアは**完全に無保証**で提供される（OSS ライセンスに準拠）。リモート接続の利用によって生じた**いかなる損害についても作者は一切責任を負わない**。

> Hub 本体は常に `127.0.0.1` にのみ bind する。リモート接続は**利用者が SSH/VPN で明示的に有効化したときだけ**成立し、その判断とリスク管理は利用者に委ねられる。

# 🚨 紛失時プレイブック（スマホを失くしたら、上から順に）

**スマホ紛失時の本当の急所は「スマホに保存された鍵そのもの」であり、Hub の認証ではない。** SSH 経路の遮断を最優先で行う。上から順に対応すること。

1. **スマホをリモートロック / ワイプ**（iPhone「探す」/ Android「デバイスを探す」）。まず端末自体を無効化する。
2. **SSH を使っていたら最優先で鍵失効**: PC/サーバーの `~/.ssh/authorized_keys` から紛失端末の公開鍵を削除（or パスワード変更）。**これでシェル経路が断たれる。** スマホに SSH が仕込まれていると、画面ロックが破られた攻撃者はシェルを取れる＝何でもできる。この経路は Hub の PIN では防げない。
3. **Hub: 「全アクセスを失効」ボタン**（設定 → リモートアクセス保護）で token を再生成する。Hub の token / 認証 cookie / PIN セッションがすべて無効化され、紛失端末を含む全デバイスが弾かれる。**このPC自身も新しいURLでの再ログインになる**点に注意。
4. **VPN: 紛失端末を外す**: Tailscale 管理画面でデバイス削除 / WireGuard は該当 peer を削除。
5. **予防策**: SSH 鍵にパスフレーズ必須・スマホ画面ロック必須・（VPN専用で SSH を置かない運用なら）Hub の任意 PIN を有効化。

> なぜ Hub 側に重い必須認証を置かないか: スマホに SSH が仕込まれている構成では Hub の認証(PIN)は無意味（攻撃者はシェルから直接何でもできる）。効くのは端末ロック・SSH鍵パスフレーズ・authorized_keys からの鍵削除。逆に **VPN だけで SSH 無しの構成**では入口が Hub だけなので、任意 PIN が「最後の扉の鍵」として意味を持つ。

## 失効ボタンと任意 PIN（設定 → リモートアクセス保護）

- **全アクセスを失効（キルスイッチ・必須機能）**: `cfg.Token` と認証 cookie 署名 secret を再生成し、既存の token URL・認証 cookie・PIN セッションをすべて無効化する。紛失時はもちろん、token 入り QR を共有してしまった等の「漏れたかも」の局面でも即押せる。SSH/VPN の鍵失効は**別途必要**（Hub token を消しても SSH 経路は塞がらない）。
- **任意 PIN（既定 OFF・希望者のみ）**: 6 桁以上の数字 PIN。設定すると**非 loopback（スマホ/VPN）アクセス時に PIN ログインを追加要求**する。loopback（このPC）は従来どおり token のみで快適。連続失敗でロックアウト（5回→1分→5分→30分の指数バックオフ）。PIN は bcrypt ハッシュで保存し平文は残さない。「VPN専用で SSH を置かない」運用で扉にもう一つ鍵をかけたい人向け。
- **新規デバイス接続の通知（SEC-C）**: 見覚えのないデバイスがリモートから接続/ログインすると、設定済みの通知（ntfy / webhook / Web Push）で本人へ知らせる。**盗まれた QR / token が使われたら即気づける**侵入検知。

## 0. これは何か

スマホや別端末から many-ai-cli の Hub（承認・進捗のダッシュボード）に接続するための手順集。Hub 画面右上の 📱ボタンから QR を出し、スマホで読み取って接続する。**URL や token を携帯で手打ちしないで済む**ようにするのが目的。

## 1. 大前提（セキュリティ）

- **Hub は常に `127.0.0.1` にだけ bind する**（外部公開しない）。これは公開保証であり変更しない。
- そのため外部端末からの到達は必ず次のいずれかの上に乗せる:
  1. **SSH ローカルフォワード**（手軽・家の中/トンネル）
  2. **VPN（WireGuard / Tailscale / Headscale）**（外出先・常時）
  3. **Docker の loopback publish**（常駐サーバー運用）
- **token は常に必須**。token 入り URL の QR は**パスワード相当**。スクショ共有・第三者に見せるのは厳禁。

## 2. 📱モバイル接続ボタンの使い方

Hub 画面右上、Usage と電源ボタン（⏻）の間の 📱 を押すとポップアップが開く。3 タブ構成。

### タブ1: URL を開く（トンネル / VPN が繋がっている時）
- `http://127.0.0.1:<port>/?token=…` の QR を表示。
- SSH トンネルまたは VPN が確立済みなら、スマホでこの QR を読むだけで Hub が開く。
- ⚠️ token 入り＝パスワード相当。

### タブ2: SSH トンネル
- スマホ SSH アプリ取り込み用の QR（`ssh://<user>@<host>:22`）と、コピー可能なコマンド `ssh -L <port>:127.0.0.1:<port> <user>@<host>` を表示。
- 前提: 接続先 PC/サーバーに OpenSSH Server が必要。
- 手順: ① スマホ SSH アプリでトンネルを張る → ② タブ1の URL QR を読む。
- ⚠️ 注記（G3）: 標準 `ssh://` QR には**ポートフォワード（`-L`）の情報は載らない**。QR でホストは取り込めても、アプリによっては `-L` を手で設定する必要がある。コマンドはコピーで貼れる。
- Windows でこの PC を接続先にする場合（G8）: 先に OpenSSH Server を有効化（`Add-WindowsCapability -Online -Name OpenSSH.Server~~~~0.0.1.0` →  `Start-Service sshd` / `Set-Service sshd -StartupType Automatic` / 22番の受信許可）。鍵認証推奨。

### タブ3: VPN（WireGuard / Tailscale 切替）
- **WireGuard**: 手元の WG クライアント設定（.conf）をテキスト欄にペースト → その場で QR 化（**サーバーに保存しない**）。WireGuard アプリで QR を読み込めばプロファイル登録完了。
- **Tailscale**: 各端末に Tailscale を入れてログイン → 同じ tailnet に参加 → タブ1の URL QR（または `http://<tailscale-ip>:<port>/?token=`）で開く。
- VPN 接続後は常にタブ1の URL QR で Hub を開く。

## 2.5. 接続方式による機能差（重要・secure context）

ブラウザは「localhost か HTTPS」でないと一部機能を許可しない。接続方式でスマホ側の使える機能が変わる。

| 接続方式 | URL 例 | Service Worker / PWA / Web Push / 音声入力(マイク) |
|---|---|---|
| **SSHトンネル** | `http://127.0.0.1:<port>` | ✅ 全部使える（localhost=secure context） |
| **VPN 直・生IP** | `http://100.x:<port>` | ❌ 使えない（secure context でない） |
| **VPN + HTTPS** | `https://名前.tailnet.ts.net` | ✅ 使える（Tailscale `tailscale serve` で https 化） |

- **フル機能で使いたいなら SSHトンネル（127.0.0.1）が最良**。Push 通知・音声入力・ホーム画面追加まで効く。
- VPN を生IPで開くと閲覧・操作はできるが Push/音声/PWA が落ちる。VPN でフル機能が欲しい場合は **Tailscale の HTTPS（`tailscale serve`）**で `https://...ts.net` にする。
- なお VPN-IP で開く場合は `allowed_hosts` 追加（§1）に加え、Hub の CSP がそのオリジンの WebSocket を許可している必要がある。

## 3. アクセスパターン早見（視覚版: access-patterns.html）

| # | 操作端末 | Hub の場所 | 経路 | 必要なもの |
|---|---|---|---|---|
| A1 | このPC | このPC | loopback 直 | なし（既定） |
| B1 | 自宅 Wi-Fi のスマホ/別PC | このPC | SSH ローカルフォワード | PC に OpenSSH Server |
| C1 | 外出先 | 自宅PC/サーバー | VPN → SSH フォワード | WireGuard 等 + SSH |
| C2 | 外出先 | 自宅 Docker 常駐 Hub | VPN → loopback publish | WireGuard + Docker |
| D1 | 自宅PC | 契約VPS（その場起動） | ランチャー SSH serve | VPS の SSH |
| D2 | 自宅PC / スマホ | 契約VPS（常駐） | ランチャー SSH tunnel | VPS 常駐 Hub |
| D3 | 外出先 / 自宅 | 契約VPS の Docker 常駐 | VPN → loopback publish | WireGuard + Docker(VPS) |

> B2（LAN へ直接 `0.0.0.0` bind）は公開保証に反するため**採用しない**。同じ目的は B1（SSHトンネル）で安全に達成する。

## 4. VPN の3方式（初心者向け）

**3つとも中身は同じ WireGuard。違うのは「鍵の名簿配りを誰がやるか」だけ。** 通信は常に端末どうしの暗号化通信で、業者にも中身は読めない。

| 方式 | ひとことで | 難しさ | 費用 | 業者を挟む | 向く人 |
|---|---|---|---|---|---|
| 素の WireGuard | VPN のエンジンを自分で全部組む | 高（鍵・ポート開放） | 無料 | 挟まない | 全部自前で握りたい |
| Tailscale | WireGuard を既製サービス化したもの | 低（ログインだけ） | 個人無料 | Tailscale 社の名簿係を経由（中身は読まれない） | とにかくラクに |
| Headscale | Tailscale の名簿係を自分の VPS で動かす無料版 | 中（VPS に構築） | 無料（VPS代のみ） | 挟まない（名簿係も自前） | ラクさと自前管理の両立（VPS持ち向き） |

たとえ話:
- 素の WireGuard … 家の鍵も配線も全部自分で工事。自由だが手間。
- Tailscale … 引っ越し業者が配線も鍵配りも代行。一番ラク。名簿は業者の事務所にある（荷物の中身＝通信は見られない）。
- Headscale … その業者の事務所を自分の敷地（VPS）に建てる。ラクさそのまま・名簿も自分の管理下。クライアントは Tailscale アプリを流用。

### Tailscale IP（100.x）の原理とセキュリティ
- ログインすると各端末に固定の仮想 IP `100.x.y.z`（`100.64.0.0/10`）が割り当てられ、tailnet 内だけで通用する。
- 役割が2層に分離: **調整サーバー=鍵の名簿配りだけ（通信内容は流れない）** / **実通信=端末どうしが WireGuard で直接 P2P 暗号化**。NAT 越え中継（DERP）を通っても中身は暗号化のまま。
- 安全性: E2E 暗号で業者も読めない。秘密鍵は端末から出ない。ログインは既存 ID プロバイダ（Google/GitHub 等）。
- 信頼の置き所: 端末を tailnet に「入れる」認可を調整サーバーに委ねる。気になるなら **Tailnet Lock**（不正ノード追加を署名で防止）か、**Headscale 自前運用**で消せる。

## 5. VPS 契約済みの人へのおすすめ順

1. まず **Tailscale**: スマホ・VPS・自宅PC の3台に入れてログイン。ポート開放不要で即繋がる。
2. 慣れて自前管理したくなったら **Headscale** を VPS に立て、クライアントはそのまま乗り換え（業者ゼロ）。
3. 完全フルコントロール派は **素の WireGuard**（ポート開放・鍵管理の手間あり）。

3台を同じ VPN に入れれば、スマホからは自宅PC・VPS 双方の Hub が「同じ家の中」に見える。各マシンが出す 📱QR を読み分けるだけで使い分けできる。

## 6. ライセンス / 商標の扱い

- WireGuard / Tailscale / Headscale は**利用者が各自導入するツールで many-ai-cli に同梱しない**ため、ライセンス表記義務はない（名前を出す nominative use は自由）。`WireGuard®` 等は各社商標。
- **同梱配布する QR 生成ライブラリは表記義務あり**。MIT/BSD/Apache-2.0/0BSD 系を選び、ライセンス文を vendor 内に保持する（GPL 系は避ける）。

## 7. トラブルシューティング

- QR を読んでも開けない → トンネル/VPN が未確立。先にタブ2(SSH) かタブ3(VPN) で接続を確立してからタブ1を読む。
- SSH で繋がらない → 接続先に OpenSSH Server が入っているか、ポート22 が通るか確認。
- 401 / token エラー → token が古い（Hub 再起動でローテート）。新しい QR を出し直す。
- VPN は繋がるが Hub が開かない → URL のホスト/ポートが対象 Hub と一致しているか（VPN 経由なら `127.0.0.1` か `100.x`/`10.8.x` か）を確認。`100.x` 等で開くなら `allowed_hosts` 追加も必要（§1）。
- Push 通知 / 音声入力 / ホーム画面追加ができない → §2.5 の secure context 問題。SSHトンネル(127.0.0.1)で開くか、VPN は HTTPS 化する。
- 保存した QR/ブックマークが急に開けない（G5）→ Hub のポートが自動退避（47778…）して URL がズレた可能性。📱から QR を出し直す。
- 外出先から自宅の素の WireGuard に繋がらない（G4）→ 自宅回線が CGNAT だとポート開放できず自宅サーバーが公開できない。Tailscale/Headscale を使う（ポート開放不要）。
