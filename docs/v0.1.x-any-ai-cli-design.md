## wrap 対象 CLI のデータ保持・規約変更リスク

`any-ai-cli` は PTY 中継に徹し、wrap 対象 CLI のネットワーク通信を傍受・改変しないため、各 CLI の利用規約・データ保持ポリシーが**そのまま**ユーザーに適用される。立て付けは CLI ごとに異なるため、ユーザーへの注意喚起もフラットに並列で示す。

### データ保持・モデル学習の立て付け（2026 年時点・要最新確認）

| CLI / バックエンド | デフォルトで学習に使われるか | opt-out / 制御 | 保持期間 |
|---|---|---|---|
| **Claude Code**（Anthropic 商用規約: API / Claude for Work / Enterprise / Education / Gov） | **使われない**（商用規約のデフォルトで除外） | opt-out 不要。エンタープライズ契約で Zero Data Retention 選択可 | API ログ最大 30 日、2025/9/14 以降は **7 日で自動削除** |
| **Codex CLI**（OpenAI: ChatGPT Plus / Pro / Business プラン経由） | **使われる可能性あり**（ChatGPT 個人プラン経由のコンテンツは学習対象になり得る） | プライバシーポータルで「Do not train on my content」、Codex Settings で「環境全体のトレーニング許可」を別途制御 | abuse 監視ログ最大 30 日、ZDR / Modified Abuse Monitoring で除外可 |
| **GitHub Copilot CLI**（GitHub: Product Specific Terms 2026/3 版） | **使われる**（プロンプトは保持され private モデルの fine-tune に利用） | 規約上の明示的な opt-out は不明（要最新規約確認） | 明示なし |

### 規約変更リスク

wrap 対象 CLI のベンダーは、第三者ツール経由のアクセスや自動化を制限する方向に規約を変更する可能性がある。その場合 `any-ai-cli` 経由の利用が違反扱いとなり得る。

- 実例: Google は 2026 年に「Gemini Code Assist を第三者ツール経由で利用することは ToS 違反」とする運用を開始し、OpenClaw / OpenCode / Antigravity 等の wrapper 利用ユーザーに対し `403 ToS` アカウント停止が多発。本ツールでは Gemini CLI を意図的に wrap 対象外とした（`docs/local/reference_provider_tos_review.md`）。
- 上表の wrap 対象 CLI についても、ベンダーが第三者自動化を制限した場合は**予告なくサポートを終了する可能性がある**。各 CLI の最新規約はユーザー責任で確認すること。

### 対応方針

1. README に上記比較表または同等の説明を載せ、各 CLI の立て付けが異なることをユーザーに明示する。
2. wrap 対象 CLI のバイナリは同梱しない。ユーザーが各ベンダーの公式手順でインストールし、各自のアカウント・契約形態を理解した上で利用する。
3. 各 CLI の最新規約はユーザー責任で確認してもらう。

## 1. 目的

`any-ai-cli` は、複数の AI コーディング CLI（Claude Code / Codex CLI）を並列で動かすときの **承認操作・進捗監視・ターミナル入出力を 1 画面のローカル Web ダッシュボードで一元管理** するツール。

単一 Go バイナリの中に以下を同梱する。

| 役割 | 内容 |
|---|---|
| Hub 本体 | `127.0.0.1` に HTTP / WebSocket サーバを起動し、ブラウザ UI と wrapper を中継する |
| wrapper | provider CLI を PTY で起動し、標準入出力を Hub に WebSocket 転送する |
| 静的 UI | xterm.js / Vanilla JS / CSS / i18n JSON を `go:embed` で同梱する |
| shell 連携 | `shell-init` で bash / zsh 用の透過ラップ関数を出力する |

## 2. 解決する課題

複数の AI CLI を並列実行すると、どのセッションが承認待ちか、どの作業が進行中か、どこで入力すべきかが分散する。`any-ai-cli` は各 CLI の TUI を壊さず PTY のまま動かし、Hub UI で次を行う。

- provider 別・プロジェクト別のセッション一覧表示
- 実行中 / 承認待ち / スタンバイ / 完了 / エラー / 切断の状態表示
- xterm.js による PTY 出力のリアルタイム描画
- Hub UI 入力欄からの PTY 入力送信
- 承認プロンプト検出時のアクションバー表示と番号送信
- 画像添付のローカル保存と provider 別 inject
- 音声入力と自動送信トリガー
- セッションログ・イベント履歴の保存
- Hub UI からの新規セッション作成

## 3. 基本方針

### 3.1 やること

- 既存 CLI のプロセスを PTY 内で起動し、CLI 自体の動作・表示・ネットワーク通信は変更しない。
- Hub はローカル WebSocket 中継に徹する。Anthropic / OpenAI API をプロキシしない。
- Hub UI は承認 UX、監視、入力、添付、設定を提供する。
- AI の承認質問を Hub UI のボタンとして検出しやすくするため、ユーザーが opt-in した場合に approval rules を Claude / Codex のグローバル指示へ注入する。
- セッションログはローカルの `~/.any-ai-cli/logs/` に保存する。
- Hub クラッシュ時は wrapper が一定時間 Hub 復帰を待ち、PTY プロセスを即死させない。

### 3.2 やらないこと

- `claude` / `codex` コマンドの PATH 乗っ取り。
- `.bashrc` / `.zshrc` 等への自動永続書き込み。
- AI 間の自動タスク分解、委譲、競合解決。
- 外部公開。Hub は `127.0.0.1` 固定で動かす。
- AI CLI の危険操作を完全に防ぐこと。
- 同一ディレクトリを複数 AI が同時編集しても安全である保証。
- 公開仕様としての Gemini CLI サポート。

## 4. 実行モデル

### 4.1 推奨起動

引数なしで `any-ai-cli` を起動すると、Hub が未起動ならそのままサーバを起動しブラウザを開く。すでに Hub が起動している場合は、既存 Hub の URL をブラウザで開いて終了する。

Windows では `any-ai-cli.exe` のダブルクリック運用を想定する。このとき開いたコンソールウィンドウは Hub サーバ本体であり、閉じると Hub は停止する。ただし走行中 wrapper は Hub クラッシュ扱いで再接続猶予に入る。

### 4.2 サブコマンド

| コマンド | 仕様 |
|---|---|
| `any-ai-cli` | Hub 起動 + ブラウザ open。起動済みなら既存 Hub を open |
| `any-ai-cli serve [--open] [--port N] [--dev] [--debug]` | Hub 単体起動。`--dev` は `web/src/` をファイルシステムから配信 |
| `any-ai-cli claude [args...]` | Hub 自動起動後、Claude Code を wrapper 経由で起動 |
| `any-ai-cli codex [args...]` | Hub 自動起動後、Codex CLI を wrapper 経由で起動 |
| `any-ai-cli wrap <provider> [args...]` | デバッグ用の明示 wrapper 起動 |
| `any-ai-cli shell-init` | bash / zsh 向け関数定義を標準出力へ出力 |
| `any-ai-cli status` | Hub の起動状態と URL を表示 |
| `any-ai-cli stop` | `%TEMP%/any-ai-cli.pid` の PID を kill |

実装上 `gemini` は `main.go` と `shell-init` に残るが、Hub UI の `/api/spawn` は `claude` / `codex` のみ許可する。

### 4.3 透過モード

`shell-init` は POSIX shell 用の関数を出力する。`ANY_AI_CLI_AUTO=1` のときだけ `claude` / `codex` / `gemini` 関数を定義する。

```bash
eval "$(any-ai-cli shell-init)"
export ANY_AI_CLI_AUTO=1
claude
codex
```

PowerShell 用スニペットは `shell-init` からは出力しない。README では `$PROFILE` に手動関数を置く運用を案内する。

## 6. Hub サーバ仕様

### 6.1 HTTP / WebSocket

Hub は `127.0.0.1:<port>` に bind する。デフォルトポートは `47777`。`NewServer` は `127.0.0.1` 固定で `http.Server` を作成する。

認証は URL query の `token` で行う。token は `~/.any-ai-cli/config.yaml` に保存され、空なら 32 byte random hex を生成する。

### 6.2 静的ファイル

通常起動では `web.FS` の `src` サブディレクトリを `go:embed` から配信する。`--dev` の場合は `web/src/` を直接配信する。

配信対象:

- `/` → `index.html`
- `/app.js`
- `/styles.css`
- `/icon.svg`
- `/i18n.js`
- `/i18n/ja.json`, `/i18n/en.json`
- `/vendor/` 配下の xterm.js と addon
- `/approval-patterns/` → `~/.any-ai-cli/approval-patterns/` の JSON ファイル

### 6.3 HTTP API

すべて token 認証付き。

| API | Method | 内容 |
|---|---|---|
| `/api/info` | GET | Hub 起動時 cwd を返す |
| `/api/spawn` | POST | UI から `wrap claude/codex` を子プロセス起動 |
| `/api/pick-directory` | POST | OS ネイティブのフォルダ選択ダイアログ |
| `/api/kill-all` | POST | 全 wrapper WebSocket を閉じる |
| `/api/shutdown` | POST | UI から Hub を停止 |
| `/api/log-config` | GET/POST | hub.log ローテーション設定とログ/添付ディレクトリ取得・保存 |
| `/api/open-dir` | POST | log または attach ディレクトリを OS ファイルマネージャで開く |
| `/api/idle-timeout` | GET/POST | UI 無接続時の wrapper kill タイムアウト分数 |
| `/api/reconnect-grace` | GET/POST | Hub クラッシュ時に wrapper が待つ秒数 |
| `/api/approval/status` | GET | approval rules opt-in 状態 |
| `/api/approval/enable` | POST | approval rules を有効化してグローバル指示へ注入 |
| `/api/approval/disable` | POST | approval rules を無効化して注入箇所を削除 |
| `/api/approval/dismiss` | POST | 初回トーストを後でにする |
| `/api/attach` | POST | multipart 画像添付 |
| `/api/slash-cmd-sources` | GET/POST | provider 別スラッシュコマンド取得元 URL / ローカルパス |
| `/api/slash-commands?provider=` | GET/POST | スラッシュコマンド一覧取得。POST は強制更新 |
| `/api/approval-patterns` | GET | provider 別 アクティブプロファイルの trigger phrase 一覧 |
| `/api/approval-patterns/<provider>` | PUT | provider の custom プロファイルを保存（official は読み取り専用） |
| `/api/approval-patterns/profile` | GET/POST | provider 別のアクティブプロファイル取得・切替（`official` / `custom`） |
| `/api/approval-patterns/copy-official` | POST | 指定 provider の official 内容を custom にコピー |

`/api/open-dir` は token に加えて remote address が loopback であることを確認し、開ける対象を `log` / `attach` に限定する。

### 6.4 起動バナー

`Server.Run` が listener を bind した直後に `startupBanner` を `os.Stdout` へ書く（`internal/hub/banner.go`）。バナーは Hub プロセスが「Web UI に連結された生命線」であることを利用者に伝えるためのもので、`internal/wrapper/wrapper.go` の `ensureHub` から `CREATE_NEW_CONSOLE` で spawn された Hub 専用コンソール窓上で表示されることを想定する（§ 17 参照）。

構成（上から順）:

1. ANY-AI ロゴ（Box-drawing 文字ベースのアスキーアート）。各セルは `█` を明色（`38;5;97` 相当）、輪郭の `╗╔╝╚║═` を黄色（`38;5;226`）で着色。
2. `Claude Code / Codex wrapper     v<version>` の 1 行。バージョンは `cmd/any-ai-cli/main.go:displayVersion` の戻り値（goreleaser ldflags `main.version` → git describe → `dev` の順で確定）。
3. `GitHub:` / `WebUI:` / `Open:` の 3 行。`Open:` には token 付き URL を出す。
4. 警告行: `WARNING: This window is connected to the Web UI. Do not close it.` を **Bold + Reverse Video + Bright Orange (`38;5;208`)** で表示。`\x1b[5m`（blink）は Windows Terminal / ConPTY / VS Code 統合ターミナル等で無視されるため使わず、reverse video で代替する。

バナーは英語表記に統一する（多言語化しない）。テストは `internal/hub/banner_test.go` で必須要素文字列と必須 ANSI コード（Bold / Reverse / BrightOrange / Reset）の存在を検証する。

## 7. wrapper 仕様

### 7.1 起動フロー

1. `any-ai-cli claude` / `codex` / `wrap <provider>` が `wrapper.Run` に入る。
2. wrapper 専用フラグを parse する。
   - 共通: `--label`, `--model`
   - Claude: `--permission-mode`
   - Codex: `--sandbox`, `--ask-for-approval`
3. `ensureHub` で Hub 起動を保証する。Hub HTTP に到達できれば既存 Hub をそのまま使い、到達できなければ自プロセスから `any-ai-cli serve` を `prepareHubSpawn`（`internal/wrapper/spawn_*.go`）経由で spawn する。Windows では `CREATE_NEW_CONSOLE`（0x10）を立てて Hub 専用のコンソール窓を新規割当し、`startupBanner`（§ 6.4）を可視化する。Unix では親ターミナルへ `stdout` / `stderr` を継承する（別ウィンドウ概念が無いため従来挙動を維持）。
4. `/ws` に接続し、`register` を送る。
5. Hub から `registered` を受け取り、session id、初期 PTY サイズ、ログパスを確定する。
6. provider CLI を PTY で起動する。
7. PTY 出力を raw log に書き、Hub へ `pty_data` 送信する。
8. Hub から `pty_input` / `pty_resize` / `attach_file` を受け、PTY に反映する。
9. PTY 終了時に `session_end` を送信する。

### 7.2 provider 引数の再構築

wrapper は Hub UI の spawn dialog から渡された安全性・モデル関連の指定を provider CLI 用の引数に戻す。

| Provider | wrapper 入力 | provider へ渡す引数 |
|---|---|---|
| Claude | `--model` | `--model <value>` |
| Claude | `--permission-mode` | `--permission-mode <value>`（`default` は渡さない） |
| Codex | `--model` | `--model <value>` |
| Codex | `--sandbox` | `--sandbox <value>` |
| Codex | `--ask-for-approval` | `--ask-for-approval <value>` |

### 7.3 PTY 入力の分割送信

Windows ConPTY と provider TUI の都合で、`text + \r` を 1 チャンクで送ると Enter として処理されない場合がある。wrapper は末尾 `\r` を検出し、本文と Enter を分割して書き込む。

- Codex は本文送信後 180ms 待ってから `\r` を送る。
- その他は 20ms 待つ。
- Claude の `@path\r` 後にテキストが続く場合は、ファイルピッカー確定後に残りのテキストが入るよう分割する。

### 7.4 Hub クラッシュ再接続

wrapper の WebSocket が切れたとき、まず Hub HTTP を probe する。

| 判定 | wrapper の動作 |
|---|---|
| Hub HTTP が生きている | 意図的切断とみなし、PTY を閉じて終了 |
| Hub HTTP に到達できない | Hub クラッシュとみなし、`wrapper_reconnect_grace_sec` 秒間、2 秒間隔で再接続を試す |
| grace が 0 | 再接続せず即 PTY を閉じる |
| grace 内に復帰 | `reattach` を送り、新しい Hub セッションとして継続 |
| grace 期限切れ | PTY を閉じる |

再接続時は wrapper 側に保持した直近 64KB の PTY 出力を base64 で渡す。Hub 側は UI 再接続用の PTY buffer に最大 512KB まで保持する。

## 9. セッション状態

Hub の状態管理は PTY 出力時刻と UI の `approval_visible` hint の組み合わせで行う。

| State | 意味 | 遷移条件 |
|---|---|---|
| `standby` | wrapper 接続済みで出力静止、承認 UI は不可視 | 初期状態、または idle 後に `approval_visible=false` |
| `running` | 直近 500ms 以内に PTY 出力あり | `pty_data` 受信 |
| `waiting` | 出力静止または承認 UI 可視 | `approval_visible=true`、または running から idle 後に hint が true |
| `completed` | provider CLI が exit code 0 相当で終了 | `session_end state=completed` |
| `error` | provider CLI が異常終了 | `session_end state=error` |
| `disconnected` | wrapper WebSocket が切断 | wrapper loop 終了、ただし completed/error 済みは維持 |

定数:

- `idleAfter = 500ms`
- `tickerInterval = 200ms`
- `uiPingInterval = 30s`
- Hub 側 PTY replay buffer = 512KB / session

UI が 0 接続になると idle timer が開始される。`hub.idle_timeout_min` 分経過すると Hub が全 wrapper WebSocket を閉じる。wrapper 側は Hub HTTP が生存しているため意図的切断として扱い、PTY を終了する。

### 10.1 レイアウト

現行 UI は添付画像 `C:/Users/admin/.any-ai-cli/attachments/4/20260511024311.jpg` の状態に近い。左にセッション一覧、右に xterm.js ターミナル、下部に承認バーと入力欄を置く。

```text
│ preview / last response    │ input + send + attach + quick commands      │
└───────────────────────────┴──────────────────────────────────────────────┘
```

### 10.2 サイドバー

- セッションを起動 cwd のプロジェクトフォルダ単位でグルーピングする。
- ヘッダーと各グループに running / waiting / standby の数を表示する。
- provider ごとの接続数を表示する。
- セッションカードには ID、provider、状態、最終応答時刻、最初/最後のユーザー入力概要、label、model、favorite、close を表示する。
- favorite と並び替えは localStorage に保存する。

### 10.3 ターミナルと入力

- xterm.js + FitAddon / WebLinksAddon / Unicode11Addon を vendored asset として使用する。
- 入力欄は複数行対応。Enter は送信、Shift+Enter は改行。
- `Ctrl+C` は選択中ならコピー、未選択なら PTY に SIGINT を送る。
- `Ctrl+D` は EOF を送る。
- `Ctrl+O` は Claude Code の折りたたみ展開に使う。
- `↑最上部へ` / `↓最下部へ` のスクロール補助を持つ。
- `/clear`, `/model`, slash picker などのクイック操作ボタンを持つ。

### 10.4 承認アクションバー

承認検出はフロントエンドで行う。xterm.js のレンダリング済みバッファ末尾を読み、approval marker / provider prompt / 番号付き選択肢を抽出する。

現行ルール:

- approval rules が有効な AI セッションは `[ANY-AI-CLI] ... [/ANY-AI-CLI]` マーカーを出す。
- Hub UI はマーカー内の YES/NO または番号付き選択肢を action-bar に変換する。
- マーカーなしの場合も provider / common の approval pattern で検出する。
- action-bar のクリック、または左右キーでフォーカスして Enter により、対応番号 + Enter を PTY へ送る。
- ターミナル側で直接番号や y/n を入力して承認が解決した場合、UI は `session_hint` を false に戻し、action-bar を消す。
- `Yes, and don't ask again` 等の永続許可系は警告色で扱う。

複数質問の一括承認:

- 1 つの `[ANY-AI-CLI]` ブロック内に複数のセクションを並べる形式を許容する。各セクションは行頭の `<数字> <空白> <質問本文>` を見出しとし、続けて `<数字>. <ラベル>` 形式の選択肢を並べる。`<数字>.` の有無で見出しと選択肢を区別する。
- Hub UI はセクションが 2 つ以上ある場合に action-bar を縦スタック表示に切り替え、各セクションごとに選択ボタン群と進捗、クリア / まとめて送信 / 閉じる ボタンを描画する。
- ユーザーが全セクションで選択を完了し「まとめて送信」を押すと、選択番号を空白区切りで連結したテキスト（例: `1 2 1 3`）+ `\r` を PTY 直送する。AI 側は `split()` で各質問の回答を取り出せる。
- キーボード操作: 入力欄が空のときは数字キーで現在フォーカスセクションを選択して次セクションへ自動前進、Tab/Shift+Tab/←/→ でセクション移動、スペースで次セクションへ移動、Enter で全選択済みなら送信。

### 10.5 音声入力

Chrome / Edge の Web Speech API を利用し、Hub UI の入力欄へ音声認識結果を挿入する。

- ボタンまたは `Alt+V` で開始/停止。
- 初回はブラウザのマイク権限が必要。
- 自動送信トリガーが有効な場合、入力末尾に設定フレーズが現れると、フレーズを除いた本文を送信する。
- 音声入力 UI には波形表示と通知音設定がある。

### 10.6 スラッシュコマンド picker

`/` ボタンまたは slash 入力から provider 別のコマンド一覧を表示する。

- Hub が markdown 取得元から `/command` と説明を抽出する。
- デフォルト取得元は `resources/slash-commands/{claude,codex}.md` の GitHub raw URL。
- 取得結果は Hub メモリ上で 24 時間 cache。
- 設定パネルで URL またはローカルファイルパスを変更できる。
- 強制更新は `/api/slash-commands` POST。

## 11. approval rules / approval patterns

### 11.1 approval rules opt-in

Hub UI の初回トーストまたは設定パネルから承認ボタン機能を有効化できる。有効化時:

1. `~/.any-ai-cli/approval-rules.md` を version 2 の内容で同期する。
2. Claude は `~/.claude/CLAUDE.md` 末尾に `@~/.any-ai-cli/approval-rules.md` を追記する。
3. Codex は `$CODEX_HOME/AGENTS.md`、未設定なら `~/.codex/AGENTS.md` に `<!-- any-ai-cli:approval-rules -->` ブロックを追記する。
4. 無効化時は該当 import / block を削除する。

rules の要点:

- セッション開始時に `ANY_AI_CLI` を確認する。
- `ANY_AI_CLI=1` の場合のみ `[ANY-AI-CLI]` マーカーを使う。
- YES/NO は `(Y:1/N:0)` を含める。
- 番号付き選択肢は `1. ... (Recommended)` を推奨選択肢にし、最後を `N. User specifies` にする。
- マーカーは確認・承認質問にだけ使う。

Hub UI から spawn した wrapper は環境変数 `ANY_AI_CLI=1` を付与される。

### 11.2 approval patterns

承認 trigger phrase は GitHub の `resources/approval-patterns/{claude,codex,common}.md` を**正本**とし、Hub 起動時にリモート fetch して `~/.any-ai-cli/approval-patterns/<provider>.official.json` を更新する。fetch 失敗時はバイナリ内ハードコード値（`internal/hub/approval_patterns.go` の `defaultApprovalPatterns`）で初期化する。TTL は 24h（次回 Hub 再起動まで再取得しない）。

ユーザー編集を保護するため、provider ごとに **official** / **custom** の 2 プロファイル制を採用する。

| ファイル | 役割 |
|---|---|
| `<provider>.official.json` | リモート md からの取得結果。読み取り専用 |
| `<provider>.custom.json` | ユーザー編集可。初期値は空（必要なら "公式からコピー" で official を写す） |
| `<provider>.json` | アクティブプロファイル内容のミラー（フロント既存ロード経路との互換） |

provider は `claude` / `codex` / `common` の 3 種類。初期 default は全 provider `official`。

#### マイグレーション

旧 `<provider>.json`（ユーザー編集を含む可能性あり）が存在する場合、Hub 起動時の `SyncApprovalPatterns` が以下を実行する：

1. `<provider>.custom.json` が無ければ、旧 `<provider>.json` を `<provider>.custom.json` にリネーム（編集内容を引き継ぐ）
2. `<provider>.official.json` が無ければハードコード値で初期化
3. 旧 `<provider>.json` が `<provider>.custom.json` と共存していたら旧を削除
4. アクティブプロファイル内容を `<provider>.json` に書き出す（既存フロントロード経路の互換）

#### リモート更新通知

起動時 fetch で `<provider>.official.json` に差分が出たら、WS イベント `approval_patterns_updated` を全 UI に broadcast する（`Providers` に差分があった provider 名のみ）。UI 側はトースト表示と、表示中の編集パネルの再ロードを行う。

#### 設定キー

| キー | 型 | 内容 |
|---|---|---|
| `approval_pattern_sources.claude/codex/common` | string | リモート md の取得元 URL（空ならビルトインデフォルト） |
| `approval_profiles.claude/codex/common` | `official` / `custom` | provider 別アクティブプロファイル |

#### UI

設定パネル「承認検出パターン」セクションには provider セレクタ・プロファイルセレクタ・「公式からコピー」ボタンが並ぶ。official 選択中はリストが読み取り専用（× ボタン・追加ボタン非表示）。custom 選択中は編集可能、初期は空のため「公式からコピー」で取り込む運用。「デフォルトに戻す」ボタンは廃止（プロファイル切替で代替）。

## 12. spawn 仕様

`/api/spawn` は Hub UI の「+ 新しいセッション」から呼ばれる。provider は `claude` / `codex` のみ。

リクエスト主要項目:

| Field | 内容 |
|---|---|
| `provider` | `claude` / `codex` |
| `cwd` | 起動ディレクトリ。空なら Hub 起動 cwd |
| `model` | provider CLI に渡す model |
| `model_selection_mode` | `auto` / `explicit` / `required` |
| `risk_confirmed` | 高リスク設定をユーザーが確認したか |
| `label` | セッションカード表示ラベル |
| `permission_mode` | Claude 用 permission mode |
| `sandbox` | Codex 用 sandbox |
| `ask_for_approval` | Codex 用 approval policy |

高リスク判定:

- Claude: model 変更、または `permission_mode=bypassPermissions`
- Codex: model 変更、または `sandbox=danger-full-access`、または `ask_for_approval=never`

高リスクかつ `risk_confirmed=false` の場合は 400 を返す。model が確定した場合は `config.yaml` の `user_prefs.spawn.last_model` に provider 別で保存する。

## 14. 設定ファイル

設定ファイルは `~/.any-ai-cli/config.yaml`。Windows では `%USERPROFILE%\.any-ai-cli\config.yaml`。

### 14.1 設定保存先の 3 分類

| 分類 | 例 | 保存先 |
|---|---|---|
| **D1: UI 表示状態**（端末ごとで自然） | theme / fontsize / lang / sidebar_width / files-tree 幅・展開・タブ / settings-section-state | **localStorage** のまま |
| **D2: ユーザー機能設定**（端末またぎたい） | voice / wake_word / trigger / notify_sound / approval_auto_switch / quick_cmds / usage_links / favorites / order / cwd_history / spawn 既定 / spawn last_model | **`config.yaml: user_prefs:`** セクション（`GET/PUT /api/user-prefs` 経由） |
| **D3: サーバ運用設定** | hub / log / approval（サーバ側有効化）/ slash_cmd_sources / approval_pattern_sources / approval_profiles / token | **`config.yaml`** のまま（直接書き換えまたは専用 API） |

D2 の値はブラウザ起動時に `GET /api/user-prefs` でミラーされ、localStorage はキャッシュとして機能する。書き換え時は `setUserPref()` で localStorage と PUT を同時に行う。初回起動時（`migrated_from_localstorage` が未設定）は localStorage 既存値をサーバへ push 移行する。

カスタム通知音のバイナリ実体は `~/.any-ai-cli/notify_sound_custom.bin` に別ファイルとして保存され、`user_prefs.notify_sound.custom_file` と `user_prefs.notify_sound.custom_mime` でそれぞれパスと MIME タイプを参照する（config.yaml 直書きによる肥大化を防ぐため）。

### 15.3 セッション JSONL

`~/.any-ai-cli/logs/sessions/<provider>_<YYYY-MM-DD_HHMMSS>_<folder>_s<id>.jsonl`

1 行 1 JSON のイベント履歴。

| Event | 内容 |
|---|---|
| `session_start` | provider, cwd, label, model, shell, pid |
| `session_reattach` | Hub 再接続時の復元情報 |
| `pty_output` | `data_b64` と ANSI 除去済み `text` |
| `user_input` | UI から送った入力と inject 有無 |
| `attach` | 添付ファイルパス、元ファイル名、provider |
| `session_end` | state, exit_code |
| `session_dismiss` | UI から削除されたセッション |

ファイル名用の provider / folder は制御文字、予約文字、空白連続などを sanitize し、80 文字で切り詰める。

### 15.4 クリーン transcript（人間可読版）

`~/.any-ai-cli/logs/sessions/<provider>_<YYYY-MM-DD_HHMMSS>_<folder>_s<id>.txt`

`.jsonl` を元に ANSI / OSC / 制御コード / スピナー行 / 連続空行 / 直前と同一の行 を除去した plain text。`internal/sessionlog/transcript.go:WriteTranscript` を `sessionlog.TranscriptPath(jsonlPath)` 宛に書く。

- **生成タイミング**: `wrapperMessageLoop` の終了処理で `Writer.Close()` を呼んだ直後（`internal/hub/server.go`）。`session_end` を経由しない `disconnected` のケースも同じ経路でカバーされる。
- **クラッシュ救済**: Hub 自体が落ちた場合は終了処理を通らないため `.txt` が作られない。`serve` 起動時に `recoverTranscripts` が `logs/sessions/*.jsonl` を走査し、対応する `.txt` が無い、もしくは `.jsonl` より古いものを遡って生成する。
- **手動生成**: `any-ai-cli log-clean <session.jsonl> -o out.txt` でも同じ変換が可能（オフライン解析用）。

## 16. セキュリティ / プライバシー

- Hub は `127.0.0.1` 固定 bind。
- token なしの HTTP API / WebSocket register は拒否する。
- `/api/open-dir` は loopback remote address と対象種別を検証する。
- `any-ai-cli` 自身は telemetry を送信しない。
- セッション raw log / JSONL にはユーザー入力と AI 出力が保存されるため、機密情報として扱う。
- slash command picker は設定された URL へ HTTPS GET することがある。
- wrap 対象 CLI は各 vendor API へ直接通信する。Hub はそれを中継・傍受しない。
- Hub UI を reverse proxy や `0.0.0.0` bind で外部公開する設計ではない。

## 21. 現行仕様上の注意

1. **承認検出は UI ヒューリスティック**
   CLI 側の表示変更、TUI のレンダリング差、言語差で検出が崩れる可能性がある。approval rules と approval patterns で補正する。

2. **dangerous / bypass 系は Hub が安全化しない**
   Claude の `bypassPermissions` や Codex の `approval=never` は provider CLI 側の承認を減らす設定であり、Hub は起動時にリスク確認を挟むだけ。

3. **Hub クラッシュ猶予は新規 wrapper に適用**
   `wrapper_reconnect_grace_sec` は wrapper 起動時に読まれるため、設定変更は既存セッションには反映されない。

4. **UI 無接続 timeout は意図的切断**
   ブラウザを閉じたまま `idle_timeout_min` を超えると、Hub が wrapper を切り、wrapper は PTY を終了する。

5. **ログは機密**
   raw log と JSONL はユーザー入力、AI 出力、添付ファイルパスを含む。

6. **同一作業ツリーの並列編集は利用者責任**
   any-ai-cli はセッションをまとめて見せるが、git conflict やファイル上書きは防がない。
